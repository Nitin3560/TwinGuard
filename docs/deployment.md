# Deployment Guide

This document describes how TwinGuard is deployed, how runtime services communicate, and why the project is organized as independent ROS 2 services rather than a single monolithic workspace.

TwinGuard was designed so that estimation, planning, supervision, navigation, and simulation can evolve independently while communicating through stable ROS 2 interfaces.

Although the current implementation primarily targets a single-machine simulation environment, the deployment architecture intentionally mirrors larger robotics software systems where multiple services communicate across a network.

---

# Deployment Architecture

TwinGuard is deployed as several independent runtime services.

```text
                          Fast DDS Discovery Server
                                      │
        ┌──────────────┬──────────────┼──────────────┬──────────────┐
        │              │              │              │              │
        ▼              ▼              ▼              ▼              ▼
    PX4 SITL     TwinGuard Runtime    Nav2      Dataset Replay   Experiment Logger
```

Each service performs one well-defined responsibility.

Communication occurs exclusively through ROS 2 topics and DDS discovery.

This separation allows individual services to be restarted, updated, or replaced without affecting the remainder of the autonomy stack.

---

# Runtime Services

## Fast DDS Discovery Server

The Discovery Server manages ROS 2 participant discovery across all runtime services.

Instead of relying on multicast discovery, every service connects to a single Discovery Server.

This approach improves reliability when running Docker containers, bridge networks, or multiple ROS 2 workspaces.

The Discovery Server performs no autonomy computation.

Its only responsibility is DDS discovery.

---

## PX4 SITL

PX4 SITL provides the simulated flight controller.

Responsibilities include:

- Vehicle dynamics
- Sensor simulation
- State estimation
- Offboard command interface
- Vehicle telemetry publication

PX4 publishes vehicle state through `px4_msgs` topics while receiving authority-scaled commands generated by the TwinGuard runtime.

PX4 remains completely unaware of localization integrity.

---

## TwinGuard Runtime

The TwinGuard Runtime contains the primary autonomy pipeline.

It executes:

- Model-based state prediction
- Residual computation
- Trust estimation
- Authority scaling
- Formation supervision
- BehaviorTree mission execution
- Offboard supervision
- Optional Kalman-based integrity estimation

This service is responsible for converting vehicle state into trust-aware autonomy decisions.

It is the only runtime responsible for publishing offboard commands to PX4.

---

## Nav2

Nav2 executes independently from the primary autonomy runtime.

TwinGuard integrates with Nav2 using:

- IsAgentTrustworthy Behavior Tree condition
- TwinGuardIntegrityLayer costmap plugin

Both plugins consume the shared `trust_state` interface.

No modifications to Nav2 planners or controllers are required.

---

## Dataset Replay

The dataset replay service injects localization degradation into live PX4 odometry.

Rather than replaying complete vehicle trajectories, it reproduces realistic localization disturbances while the remainder of the autonomy stack executes normally.

Keeping replay independent allows identical autonomy software to be evaluated using either live simulation or recorded degradation scenarios.

---

## Experiment Logger

The experiment logger records ROS 2 topics and runtime diagnostics.

Separating logging from the autonomy runtime prevents recording workloads from affecting deterministic control execution.

Generated artifacts include:

- ROS bags
- CSV metrics
- Runtime diagnostics
- Validation plots

# Runtime Communication

Every control cycle begins with fresh PX4 telemetry.

```text
PX4 SITL
      │
VehicleOdometry
      │
      ▼
Model-Based State Prediction
      │
Residual Computation
      │
Trust Estimation
      │
trust_state
      │
 ┌────┴─────────────┐
 │                  │
 ▼                  ▼
Behavior Tree     Nav2
 │                  │
 └──────┬───────────┘
        ▼
Formation Supervisor
        │
Authority Scaling
        │
PX4 Offboard Interface
        │
        ▼
PX4
```

Every iteration begins with newly received vehicle state.

Control outputs never feed directly back into the integrity estimator.

Localization confidence is always computed from fresh observations rather than previous control decisions.

# Docker Deployment

TwinGuard provides Docker support for reproducible development and deployment.

Separate containers are provided for:

- PX4 SITL
- ROS 2 workspace
- TwinGuard runtime
- Nav2
- Dataset replay
- Experiment logging

Every container connects to the Fast DDS Discovery Server while remaining operationally independent.

This deployment model allows individual services to be restarted without interrupting unrelated runtime components.

---

# Recommended Startup Sequence

For a clean runtime initialization, services should be launched in the following order.

1. Fast DDS Discovery Server
2. PX4 SITL
3. Micro XRCE-DDS Agent
4. TwinGuard Runtime
5. Nav2 (optional)
6. Dataset Replay (optional)
7. Experiment Logger

After initialization, TwinGuard immediately begins consuming PX4 vehicle state and publishing trust-aware supervisory outputs.

---

# Design Rationale

The deployment architecture follows the same design principles used throughout TwinGuard.

- Each service performs one responsibility.
- Runtime interfaces remain stable.
- Components communicate exclusively through ROS 2 topics.
- Safety-critical autonomy remains isolated from supporting infrastructure.
- Services can evolve independently without architectural redesign.

This modular deployment strategy improves maintainability, simplifies debugging, and provides a natural path toward future distributed multi-UAV deployments while remaining lightweight enough for local PX4 SITL development.

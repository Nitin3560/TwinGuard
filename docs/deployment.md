# Deployment Guide

This document explains how TwinGuard is deployed and how the different runtime services communicate.

Unlike a traditional ROS 2 workspace where every component runs in a single environment, TwinGuard separates major subsystems into independent services. This keeps safety-critical autonomy isolated from supporting infrastructure while making the system easier to debug, restart, and extend.

---

# Deployment Overview

TwinGuard is deployed as four independent runtime services.

```
                   Fast DDS Discovery Server
                              │
      ┌──────────────┬─────────┴─────────┬──────────────┐
      │              │                   │              │
      ▼              ▼                   ▼              ▼
 PX4 SITL      TwinGuard Runtime      Nav2 Stack    Experiment Logger
```

Each service has a single responsibility and communicates using ROS 2 DDS discovery.

---

# Runtime Services

## Discovery Server

The Fast DDS Discovery Server is responsible for service discovery between all ROS 2 containers.

Instead of relying on multicast discovery, every container connects to a central discovery server.

This approach is significantly more reliable when running Docker containers on bridge networks.

---

## PX4 SITL

PX4 SITL provides the simulated flight controller.

Responsibilities:

- Vehicle dynamics
- Sensor simulation
- Vehicle state estimation
- Offboard interface

PX4 publishes vehicle state and receives offboard commands generated by TwinGuard.

---

## TwinGuard Runtime

This is the primary autonomy container.

It runs:

- Integrity estimation
- Digital Twin
- Trust scoring
- Formation supervisor
- Behavior Tree
- Offboard supervisor
- EKF integrity pipeline (optional)

This service is responsible for producing all trust-aware autonomy decisions.

---

## Nav2

Nav2 runs independently from the autonomy runtime.

TwinGuard extends Nav2 through:

- IsAgentTrustworthy BT plugin
- TwinGuardIntegrityLayer costmap plugin

The navigation stack remains unchanged while automatically consuming localization integrity.

---

## Experiment Logger

The logger records ROS 2 bags during experiments.

Keeping logging separate prevents recording workloads from affecting runtime autonomy performance.

---

# Communication

The deployment relies on ROS 2 topics and PX4 messages.

Main runtime flow:

```
PX4
    │
VehicleOdometry
    │
TwinGuard Runtime
    │
trust_state
    ├────────────► Nav2
    │
PX4 Offboard Interface
```

Every runtime iteration begins with fresh PX4 state information.

---

# Docker Architecture

The project contains separate Docker environments for:

- PX4 SITL
- ROS 2 Workspace
- Runtime services

Each container starts independently and connects through the Discovery Server.

This modular design makes individual services easier to restart without affecting the rest of the system.

---

# Deployment Order

The recommended startup order is:

1. Start Fast DDS Discovery Server
2. Launch PX4 SITL
3. Launch TwinGuard Runtime
4. Launch Nav2
5. Start experiment logging (optional)

Once all services are running, TwinGuard begins receiving PX4 odometry and publishing trust-aware offboard commands.

---

# Why This Deployment

Separating the runtime into independent services provides several advantages.

- Fault isolation
- Easier debugging
- Independent restarts
- Cleaner ROS 2 networking
- Better scalability for future distributed deployments

The architecture also mirrors how larger robotics systems are commonly deployed, making it easier to extend TwinGuard beyond a single-machine simulation.
